Technical Analysis and Remediation of Display Subsystem Failures in Integrated ESP32-S3 HMI Platforms: The Waveshare 3.5B Case Study
1. Introduction: The Complexity of Modern Embedded HMI
The evolution of embedded systems has witnessed a paradigm shift from discrete, microcontroller-based designs to highly integrated System-on-Chip (SoC) architectures capable of driving sophisticated Human-Machine Interfaces (HMI). The Espressif ESP32-S3 series, with its dual-core Xtensa LX7 architecture and vector instructions, sits at the forefront of this transition, enabling low-cost, high-performance graphical displays. However, this integration introduces significant complexity in hardware abstraction layers. The reported failure—a persistent "black screen" and non-functional backlight on a Waveshare ESP32-S3-Touch-LCD-3.5B despite successful firmware flashing—serves as a paradigmatic example of the friction between generic driver implementations and specialized hardware topologies.
The failure to illuminate the backlight or render pixels is not merely a symptom of a "bug" but a fundamental misalignment between the logical driver configuration and the physical reality of the device's interconnects. In generic development boards, a display backlight is often controlled directly via a Pulse Width Modulation (PWM) capable General Purpose Input/Output (GPIO) pin, such as GPIO 21. However, the Waveshare "Type B" variant represents a more advanced class of HMI hardware where pin conservation is paramount.1 To support high-bandwidth Quad SPI (QSPI) interfaces for the display and parallel interfaces for cameras, control signals are offloaded to secondary controllers.
This report establishes that the "No Backlight" condition is caused by the firmware's failure to address the TCA9554 I/O Expander, which acts as the gatekeeper for the display's power and reset signals. Furthermore, the visual data failure stems from utilizing a standard SPI driver (ILI9486) for a display wired and configured for QSPI communication via the AXS15231B controller. The following analysis provides an exhaustive deconstruction of the hardware architecture, the communication protocols involved, and the specific firmware engineering required to remediate these failures.

2. Hardware Architecture Analysis: The "Type B" Distinction
To engineer a functional solution, one must first rigorously define the hardware substrate. The user's insistence on possessing the "3.5B" variant is the critical diagnostic datum. Waveshare produces multiple 3.5-inch display modules, and the suffix "B" denotes a radical departure in internal architecture compared to the standard model.
2.1 The ESP32-S3R8 Computational Core
At the heart of the Waveshare ESP32-S3-Touch-LCD-3.5B is the ESP32-S3R8 module.2 The "R8" designation indicates the presence of 8 MB of Octal Pseudo-Static RAM (PSRAM) integrated into the package. This memory is not a luxury; it is a structural necessity for the display subsystem.
A standard 320x480 display with a 16-bit color depth (RGB565) requires 307,200 bytes of memory for a single frame buffer.




$$\text{Buffer Size} = 320 \times 480 \times 2 \text{ bytes} \approx 300 \text{ KB}$$


While the ESP32-S3 includes 512 KB of internal SRAM, much of this is reserved for the Wi-Fi stack, Bluetooth controller, and FreeRTOS heap. Allocating a full frame buffer in internal SRAM is often impossible without destabilizing the system. The 8 MB Octal PSRAM allows for double-buffering (ping-pong buffers), which is essential for tear-free animation and high-performance GUIs like LVGL. The firmware must be compiled with specific flags to enable "OPI PSRAM" mode, or the display initialization will fail due to memory allocation errors before a single pixel is drawn.
2.2 The Peripheral Control Bus (I2C)
The most significant architectural feature of the "Type B" board—and the direct cause of the backlight failure—is the abstraction of control signals. To conserve the high-speed pins on the ESP32-S3 for the QSPI display bus and the DVP camera interface, Waveshare engineers routed low-speed control signals through an I2C bus.1
The I2C (Inter-Integrated Circuit) bus functions as a control plane, distinct from the data plane used for pixels.
* SDA (Serial Data): GPIO 8
* SCL (Serial Clock): GPIO 9
On this bus, several critical devices reside. The firmware must explicitly address these devices to bring the board to life.
1. TCA9554 (IO Expander): Address 0x20. This chip controls the reset lines and the backlight.
2. GT911/FT6336 (Touch Controller): Address 0x14 or 0x5D.
3. AXP2101 (Power Management): Address 0x34. This chip controls the voltage rails.
The user's previous attempt to toggle GPIO 21 fails because GPIO 21 is not electrically connected to the backlight transistor. It is likely routed to the camera header or the LCD data bus. The backlight control signal is physically located on Pin 10 (P2) of the TCA9554 chip. Therefore, "turning on the backlight" is not a GPIO operation; it is a communication protocol operation involving the transmission of register addresses and bitmasks over I2C.
2.3 The Display Subsystem (QSPI)
The display panel utilizes an In-Plane Switching (IPS) matrix driven by the AXS15231B controller.2 This controller is specifically designed for high-resolution, high-refresh-rate applications and supports the Quad SPI (QSPI) interface.
Comparison of Protocols:
* Standard SPI (ILI9486): Transmits data on a single line (MOSI). It requires 1 clock cycle per bit.
* Quad SPI (AXS15231B): Transmits data on four parallel lines (D0, D1, D2, D3). It transmits 4 bits per clock cycle.
The physical mismatch is catastrophic for communication. If the firmware utilizes an ILI9486 driver, it sends commands assuming a single data line. The AXS15231B, wired for QSPI, monitors four data lines. It will interpret the single-line SPI signal as malformed data or noise. Even if the command structure were compatible (which it is not), the physical layer mismatch ensures that the display controller never receives a valid initialization sequence, resulting in the observed black screen.
2.4 Power Management (AXP2101)
A frequently overlooked component in "black screen" troubleshooting is the AXP2101 Power Management IC (PMIC). Unlike simple Low Dropout (LDO) regulators that are "always on," the AXP2101 is a programmable power computer.4 It manages multiple voltage rails (DLDO, ALDO, etc.).
* Risk Factor: If the default power-on state of the AXP2101 disables the ALDO3 or BLDO1 rails (commonly used for display logic and backlight power), the display driver IC will remain unpowered. No amount of correct driver code can initialize a chip with 0 Volts at its VCC pin. The firmware must include an initialization routine for the AXP2101 to explicitly enable these rails.

3. The Backlight Failure Mechanism: TCA9554 Deep Dive
The symptom "NO BACKLIGHT" is the most diagnostically valuable piece of evidence. In embedded systems, the backlight circuit is typically a simple high-side switch or low-side N-channel MOSFET driven by a logic signal. The failure of the user's attempt to force GPIO 21 HIGH confirms that the control signal path is not direct.
3.1 TCA9554 Functional Description
The TCA9554 is a remote 8-bit I/O expander for the I2C bus. It consists of a communication interface, a state machine, and a set of internal registers that control the direction and state of 8 quasi-bidirectional I/O pins (P0–P7).1
To control a pin on the TCA9554, the ESP32 master must perform a multi-step handshake:
1. START Condition: Initiate communication on the I2C bus.
2. Addressing: Send the 7-bit device address (0x20) plus the Write bit.
3. Register Selection: Send the address of the internal register to be modified.
4. Data Transmission: Send the data byte to define the state of the pins.
5. STOP Condition: Terminate the transaction.
3.2 Pin Mapping on Waveshare 3.5B
The documentation for the Waveshare ESP32-S3-Touch-LCD-3.5B identifies the specific mapping of the TCA9554 pins 1:
* EXIO0 (P0): LCD_RST (LCD Reset). Active Low.
* EXIO1 (P1): TP_RST (Touch Panel Reset). Active Low.
* EXIO2 (P2): LCD_BL (Backlight Control). Active High.
* EXIO3 (P3): SD_CS (SD Card Chip Select).
* EXIO4 (P4): USB_SEL (USB Function Selection).
* EXIO5 (P5): SPK_EN (Speaker Enable).
* EXIO6 (P6): AUDIO_PWR (Audio Power).
The Failure Chain:
1. Default State: Upon power-up, the TCA9554 pins default to inputs (high impedance).
2. Circuit Behavior: With P2 floating (high impedance), the pull-down resistor on the MOSFET gate pulls the signal to Ground (0V).
3. Result: The backlight circuit remains open (OFF).
4. User Action: The user toggles GPIO 21 on the ESP32. This changes the state of a disconnected pin inside the SoC but has zero influence on the TCA9554, which is waiting for I2C commands.
5. Observation: The screen remains dark.
3.3 Required Remediation: I2C Sequence
To activate the backlight, the firmware must execute a specific sequence of I2C operations. The TCA9554 has a Configuration Register (0x03) that controls pin direction (1=Input, 0=Output) and an Output Port Register (0x01) that controls logic level.
Correct Initialization Logic:
1. Configure P0, P1, P2 as Outputs:
   * Target Register: 0x03 (Configuration).
   * Value: 0xF8 (Binary 11111000).
   * Explanation: Bits 0, 1, and 2 are set to 0 (Output). Bits 3-7 are set to 1 (Input/Default).
2. Set P0, P1, P2 High:
   * Target Register: 0x01 (Output Port).
   * Value: 0x07 (Binary 00000111).
   * Explanation: Bit 2 (Backlight) is HIGH. Bit 0 (LCD Reset) and Bit 1 (Touch Reset) are HIGH (De-asserting reset).
This sequence is the only way to electrically close the backlight circuit. Until this I2C transaction occurs, the screen will remain physically incapable of emitting light.

4. The Display Subsystem: AXS15231B and QSPI
Resolving the backlight issue is only the first step. Once illuminated, the screen will likely display black pixels because the display controller has not been initialized. The user's selection of the ILI9486 driver is architecturally incompatible with the AXS15231B QSPI controller.
4.1 Bandwidth and Protocol Physics
The choice of QSPI for the "Type B" board is driven by the physics of data transmission.
* Resolution: 320 x 480 pixels.
* Color Depth: 16-bit (2 bytes per pixel).
* Frame Size: 307,200 bytes (approx. 2.45 Mb).
Standard SPI (ILI9486):
* Transmits 1 bit per clock cycle.
* At a standard 40 MHz clock, the theoretical maximum throughput is 40 Mbps.
* Frame time: $2.45 \text{ Mb} / 40 \text{ Mbps} = 61.25 \text{ ms}$.
* Max FPS: ~16 FPS.
Quad SPI (AXS15231B):
* Transmits 4 bits per clock cycle (D0, D1, D2, D3).
* At 40 MHz, the throughput is 160 Mbps.
* Frame time: $2.45 \text{ Mb} / 160 \text{ Mbps} = 15.3 \text{ ms}$.
* Max FPS: ~65 FPS.
Using an ILI9486 driver forces the ESP32 to serialize data onto a single pin (MOSI). The AXS15231B, configured for QSPI, samples four pins simultaneously. It will interpret the single-line data stream as fragmented, nonsensical commands.
4.2 Hardware Pin Definitions
The firmware must define the correct GPIOs for the QSPI bus. Based on the Waveshare ESP32-S3-Touch-LCD-3.5B schematic and pin definition 3, the mapping is:

4.3 AXS15231B Initialization Sequence
The AXS15231B requires a complex initialization sequence to configure its internal charge pumps, gamma curves, and power settings. Crucially, it uses a vendor-specific "Unlock" command to access extended registers.2
Critical Command Sequence:
1. Unlock Register (0xFF): Data 0x20, 0x10, 0x10. This unlocks the restricted commands.
2. Digital Gamma (0xBB): Sets the VCOM voltage to match the specific IPS panel physics. Incorrect values here result in a washed-out or flickering image.
3. Pixel Format (0x3A): Data 0x55. Sets the interface to 16-bit color (RGB565).
4. Sleep Out (0x11): Wakes the internal oscillator. Requires a 120ms delay.
5. Display On (0x29): Activates the gate drivers.
If the user employs the ILI9486 driver, none of these specific commands (especially the 0xFF unlock) are sent. The controller remains locked in its default state, unable to process pixel data.

5. Comprehensive Remediation Roadmap
To restore functionality, the firmware architecture must be refactored to align with the "Type B" hardware reality. This requires a layered approach: Power Layer, Control Layer (I2C), and Data Layer (QSPI).
Phase 1: Environment Configuration
The development environment (Arduino IDE or PlatformIO) must be configured to support the high-performance memory architecture.
* Board: ESP32-S3 Dev Module.
* Flash Mode: QIO 80MHz.
* PSRAM: OPI PSRAM (Octal Mode). Critical: If this is disabled or set to QSPI PSRAM, the 8MB buffer cannot be accessed, causing immediate crashes during display initialization.
* Partition Scheme: Huge APP (3MB No OTA/1MB SPIFFS) to accommodate large graphical libraries like LVGL.
Phase 2: Power and Peripheral Initialization (PowerManager)
Before initializing the display object, the setup() function must prepare the physical layer.
1. I2C Initialization:
C++
Wire.begin(8, 9); // SDA=8, SCL=9

2. AXP2101 Configuration:
   * Check presence at address 0x34.
   * Command AXP2101 to enable ALDO1, ALDO2, ALDO3, ALDO4, BLDO1, BLDO2. While not all are used, enabling the logic rails ensures the AXS15231B core voltage (1.8V) and analog voltage (3.3V) are present.
   3. TCA9554 Configuration (Backlight & Reset):
   * Write 0xF8 to Register 0x03 (Config) to set P0, P1, P2 as Outputs.
   * Write 0x00 to Register 0x01 (Output) to pull Reset LOW (Hard Reset).
   * Wait 100ms.
   * Write 0x07 to Register 0x01 (Output) to pull Reset HIGH and Backlight HIGH.
   * Wait 200ms for the display controller to stabilize after reset.
Phase 3: Display Driver Implementation
The firmware should utilize a library capable of QSPI. Arduino_GFX (by MoonOnOurNation) is the industry standard for this. It includes a specific driver for the AXS15231B.
Driver Instantiation (Pseudo-Code):


C++




#include <Arduino_GFX_Library.h>

// Step 1: Define the Data Bus (QSPI)
Arduino_DataBus *bus = new Arduino_ESP32QSPI(
   42 /* CS */, 41 /* SCK */,
   1 /* D0 */, 2 /* D1 */, 3 /* D2 */, 4 /* D3 */
);

// Step 2: Define the Display Panel
Arduino_GFX *gfx = new Arduino_AXS15231B(
   bus, 
   GFX_NOT_DEFINED, /* RST - Handled manually via TCA9554 */
   0, /* Rotation */
   true, /* IPS Mode */
   320, /* Width */
   480  /* Height */
);

By passing GFX_NOT_DEFINED for the reset pin, we prevent the library from attempting to toggle a non-existent GPIO, relying instead on the manual TCA9554 reset performed in Phase 2.
Phase 4: Verification
Upon flashing this corrected firmware, the boot sequence will be:
   1. Serial Log: I2C Bus Initialized.
   2. Serial Log: TCA9554 Configured. Backlight Enabled. -> Visual: Backlight turns on (screen glows black/grey).
   3. Serial Log: AXS15231B Init Start.
   4. Visual: Screen flickers briefly as gamma settings load, then displays the test color (Cyan).

6. Advanced Considerations: LVGL and Frame Buffering
For a production-grade HMI, simply filling the screen with color is insufficient. The goal is to run a GUI framework like LVGL.
6.1 Direct Memory Access (DMA)
Using QSPI with the ESP32-S3 allows for DMA transfers. This means the processor can calculate the next frame while the DMA engine pushes the current frame to the display. To enable this:
   * The frame buffer must be allocated in DMA-capable memory.
   * For full-frame buffering (recommended for 320x480), the buffer must be in PSRAM.
   * Use heap_caps_malloc(size, MALLOC_CAP_SPIRAM) to ensure the buffer resides in the 8 MB Octal RAM.
6.2 Tear Effect (TE) Synchronization
The board routes the TE signal to GPIO 40.3 This signal pulses when the display controller is in the vertical blanking interval (VBI). To prevent "tearing" (where the display updates midway through a refresh), the firmware should utilize this signal.
   * Implementation: Configure GPIO 40 as an interrupt input.
   * Logic: Only trigger the DMA transfer of the new frame buffer when the TE interrupt fires. This synchronizes the ESP32's write pointer with the display's read pointer.

7. Conclusion
The diagnostic journey from a "black screen" to a functional HMI on the Waveshare ESP32-S3-Touch-LCD-3.5B illustrates the critical importance of hardware-software alignment. The user's initial failure was not due to a hardware defect but resulted from treating a specialized, highly integrated Type-B device as a generic, discrete component.
The specific failure points were:
   1. Backlight Control: The expectation of direct GPIO control ignored the TCA9554 I/O Expander (Address 0x20, Pin P2), which physically gates the backlight power.
   2. Display Protocol: The use of an SPI ILI9486 driver was incompatible with the QSPI AXS15231B controller, causing a total breakdown in data transmission.
   3. Power Sequencing: The likely omission of AXP2101 configuration left the display logic unpowered.
By adopting the remediation strategy outlined in this report—specifically, implementing the I2C control layer for the TCA9554 and utilizing the correct QSPI driver topology—the device will function as designed, unlocking the full potential of the ESP32-S3R8's high-speed interface and substantial memory resources.
