3.0 Hardware Architecture: The Phenix Phantom
The realization of these theoretical protocols requires a hardware platform of exceptional density and versatility. The Phenix Phantom is built around the Waveshare ESP32-S3 Touch LCD 3.5" (Type B), a carrier board that integrates the computational core, display driver, and power management into a single compact unit.
3.1 The Computational Core: ESP32-S3R8
The heart of the Phantom is the ESP32-S3R8 System-on-Chip (SoC). This dual-core Xtensa LX7 processor, running at 240 MHz, provides the necessary headroom for the "Real-Time" requirements of the system.
* Core 0 (The Whale): Dedicated to the LoRa radio stack and the "Whale Song" transport protocol. It handles the precise timing of Chirp Spread Spectrum (CSS) modulation, ensuring signal integrity even below the noise floor.3
* Core 1 (The Click): Dedicated strictly to the UI and Haptics. This separation ensures that tactile feedback is instantaneous (<10ms latency), preventing the sensory dissonance that breaks the illusion of physical controls.3
3.2 The Bandwidth of Illusion: QSPI vs. SPI
The selection of the "Type B" variant of the Waveshare board is a rigid engineering requirement. Standard embedded displays typically use a 4-wire SPI interface, which is limited to bandwidths that support frame rates of only 15-20 FPS. This is insufficient for the "Phantom Mechanisms"—virtual knobs and sliders—which require fluid, 60 FPS animation to trick the brain's proprioceptive loop.5
The Type B board utilizes the AXS15231B controller with a QSPI (Quad SPI) interface. By using four data lines (D0-D3) instead of one, the theoretical bandwidth increases fourfold, sustaining the ~73 Mbps throughput required for a 320x480 display at 16-bit color depth.5 This high frame rate is the "Bandwidth of Illusion" necessary to make a sheet of glass feel like a mechanical machine.
3.3 The Memory Hierarchy: Octal PSRAM (OPI)
To support the QSPI display and the AI audio buffers, the system relies on 8MB of Octal PSRAM (OPI). Unlike standard Quad PSRAM, Octal PSRAM utilizes an 8-bit wide bus for data transfer. This high-speed memory is critical for Double Buffering:
1. Buffer A: The CPU renders the next frame of the UI (e.g., the rotation of the virtual encoder) into Buffer A in PSRAM.
2. Buffer B: The DMA engine pushes the previously rendered Buffer B to the display via the QSPI bus.
This parallel operation eliminates screen tearing and flickering, enabling complex alpha-blended effects like the "Glow" indicators for the Universal Language UI.5
3.4 The Octal PSRAM vs. SPI Conflict: A Critical Resolution
A major engineering challenge identified in the design phase is the conflict between the Octal PSRAM bus and the standard SPI GPIO mappings on the ESP32-S3. The Octal PSRAM interface consumes GPIOs 33, 34, 35, 36, and 37 for its high-speed data lines (SPID4-SPID7).
   * The Conflict: Legacy designs often map the LoRa radio (an SPI device) to these same pins (CS on 34, MOSI on 35, etc.). Connecting an external peripheral to the PSRAM bus lines causes immediate system instability, memory corruption, and boot failures.4
   * The Resolution: The Phenix Phantom Pinout strictly isolates the Radio bus from the PSRAM bus. It utilizes the "Native FSPI" pin mapping where possible on the rear header or reclaims GPIOs freed by the removal of mechanical switches.

4.0 Firmware Architecture: The 'Sorcery' AI Integration
The firmware for the Phenix Phantom is a sophisticated fork of the xiaozhi-esp32 repository, re-architected to support the autonomous "Sorcery" agent. It utilizes the FreeRTOS kernel to manage the complex interplay of real-time audio, network streaming, and user interface rendering.
4.1 Memory Partitioning and Optimization
The system operates within the constraints of the ESP32-S3's memory map. The firmware utilizes the v2 partition scheme, which requires a full flash erase for migration.8
      * NVS (Non-Volatile Storage): This partition stores the device's cryptographic identity, Wi-Fi credentials, and the 'Sorcery' API token. Crucially, it secures the credentials for the custom provisioning flow, removing dependence on external portals.8
      * SPIFFS (File System): The standard SPIFFS partition is resized (enlarged) at the expense of the app partition. This is necessary to store the larger, custom wake-word models ("Hey Phenix") and the high-resolution graphical assets required for the Wonky Sprout UI.8
      * RAM Allocation Strategy:
      * 512KB Internal SRAM: This is the scarce resource. It is reserved for DMA descriptors and critical FreeRTOS task stacks.
      * 8MB Octal PSRAM: Used for "heavy lifting." The Audio Ring Buffer (20KB-50KB) and LVGL Display Buffers (approx. 150KB) are allocated here using MALLOC_CAP_SPIRAM.8
      * Double-Buffering: Because the ESP32's DMA controller cannot access PSRAM directly, audio data flows through a pipeline: Network $\to$ PSRAM Ring Buffer $\to$ memcpy() $\to$ Internal SRAM DMA Buffer $\to$ I2S Peripheral. This introduces CPU overhead but is required for stability.8
4.2 FreeRTOS Task Hierarchy
The firmware uses an event-driven architecture managed by a central Application class.
      * SorceryProtocol Task: A high-priority task responsible for the Secure WebSocket (WSS) connection. Due to the heavy overhead of TLS handshakes and JSON parsing (for the MCP protocol), this task is allocated a larger stack size of 8KB–10KB.8
      * lv_timer_handler Task: Dedicated to the LVGL graphics library. It must run frequently enough to maintain 30+ FPS for the UI animations.
      * Audio Pipeline Tasks: These tasks manage the flow of data between the microphone, the Opus encoder, and the network. They run across both cores to balance the load.8
4.3 The 'Sorcery' AI Pipeline
The 'Sorcery' agent is not a simple chatbot; it is a multimodal entity integrated into the hardware abstraction layer (HAL).
      * Acoustic Front End (AFE):
      * Input: INMP441 MEMS Microphone. It uses an I2S interface to output digital audio directly, bypassing the noisy internal ADC of the ESP32 for a lower noise floor.8
      * Output: ES8311 Codec. Configured via I2C and fed via I2S.
      * Full-Duplex Operation: The I2S bus is configured for simultaneous input and output (I2S_DIN and I2S_DOUT). This supports "Barge-In" functionality, allowing the user to interrupt the agent while it is speaking—a critical feature for natural conversation.8
      * Communication Stack:
      * Transport: Secure WebSockets (WSS). Chosen over MQTT for its native support in the ESP-IDF and easier firewall traversal.
      * Protocol: Model Context Protocol (MCP). This allows the AI agent to "perceive" the device's context (battery level, IMU orientation) and "act" upon it (triggering haptic effects, changing LED colors) via structured JSON-RPC calls.8
      * Latency: The pipeline is optimized for a "Time-to-Ear" latency of <800ms.8
