2.1.1 The Icosahedron as the Primitive
The geodesic dome begins as an IcosahedronGeometry. To make each "pane" (face) interactive, the geometry must be treated not as a single mesh, but as a collection of addressable components. Standard IcosahedronGeometry in Three.js shares vertices between faces to optimize rendering performance. However, this creates a topological problem for the "Cognitive Shield": clicking on a vertex affects multiple faces, blurring the boundaries of the "interactive elements."
* Topological Deconstruction: To make faces interactive individually (e.g., clicking a specific triangle to open a modal), the geometry must be "exploded" or "non-indexed." This separates the vertices so that each triangle has its own unique set of 3 vertices. The logic involves iterating through the geometry faces and creating a unique mesh for each, or using a BufferGeometry where vertices are duplicated.4
* The Component Model: In R3F, the dome is not a single object but a generated array of <GeodesicFace /> components. This allows each face to hold its own state (entropy level, content type, color).
