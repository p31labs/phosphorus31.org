The engineering solution to a failed Wye circuit is a transformation to a "Delta" (Mesh) topology. A Delta connection has no neutral point; the phases are connected end-to-end in a triangle. 3. Browser Extension Architecture: Securing the Digital Perimeter
The primary interface for receiving and managing these funds is a custom Chrome Extension. 3.1 Manifest V3 (MV3) Compliance and Security Patterns
The transition to Manifest V3 represents a fundamental shift in extension security, designed to address critical concerns around security, performance, and user privacy.8 While often criticized for limiting ad-blockers, MV3's constraints align with the Operator's "Sovereign" philosophy by forcing a cleaner, more isolated architecture.
Service Workers as Ephemeral Guardians:
MV3 replaces persistent background pages with Service Workers.9 These are event-driven scripts that run only when needed and terminate when idle. This significantly reduces the attack surface. In a persistent background page (MV2), sensitive variables (like decrypted keys) might linger in memory for the duration of the browser session, vulnerable to memory scraping attacks. In MV3, the service worker's ephemeral nature forces the developer to adopt a "stateless" security model.9 Keys cannot be stored in global variables; they must be retrieved from secure storage (encrypted) only at the moment of signing and wiped immediately after. This aligns with the "Deep Work" philosophy of the Operator, preventing "voltage spikes" of continuous exposure.7
Prohibition of Remote Code:
MV3 strictly prohibits the execution of remotely hosted code.9 An extension can only execute JavaScript that is included within its package and subject to review by the Chrome Web Store. This is a critical defense against "Man-in-the-Middle" attacks or the injection of malicious scripts by a compromised server. For the Operator, this means the "Genesis Gate" protocol logic is immutable and auditable. It prevents the "Banal Deception" of a system that changes its rules without the user's consent.1 The extension becomes a fixed, trustworthy tool—a digital hammer—rather than a malleable service.
Content Security Policy (CSP):
MV3 introduces stricter Content Security Policy requirements.8 The CSP must be structured as an object with specific directives, preventing the execution of arbitrary strings as code (e.g., eval()). This protects the wallet interface from Cross-Site Scripting (XSS) attacks, ensuring that a malicious website cannot inject code to siphon the donation stream.

Auditability and Minimal Surface Area:
micro-eth-signer is a minimal library for Ethereum transactions, addresses, and smart contracts. It relies on noble-curves for its cryptography, which is an independently audited, high-security library.10 The codebase is small (approx. 29KB gzipped), "tree-shakeable," and contains zero dependencies beyond the noble suite. This makes it easily auditable by the Operator. In a forensic scenario (court case), the Operator can point to the specific lines of code that govern the wealth, proving that there are no backdoors or hidden dependencies—a crucial factor in the "Centaur Defense" against accusations of being a "fake".1
Network Independence:
Crucially, micro-eth-signer contains "no network code".10 Unlike libraries like ethers.js or web3.js that often bundle network providers (like Infura or Alchemy), micro-eth-signer separates the signing of transactions from their broadcasting. The extension can route transactions through a custom, privacy-preserving RPC (e.g., a local Erigon node, a specific privacy proxy, or a Tor relay) without the library leaking IP data to third-party infrastructure.10 This is essential for preventing IP logging that could link the Operator's physical location to the donation wallet.12

4. Privacy Engineering: The Stealth Address Implementation (ERC-5564)
The "Floating Neutral" crisis demands that the Operator's income streams remain invisible to adversarial surveillance. Standard Ethereum transactions are public, allowing any observer to trace the flow of funds from donor to recipient. To solve this, the "Donation Wallet" will implement ERC-5564 (Stealth Addresses), a standard for private, non-interactive transfers.
4.1 Cryptographic Mechanics of Invisible Donations
Stealth addresses allow the Operator to publish a single "Meta-Address" (e.g., on a website or social media) while ensuring that every incoming donation lands in a unique, unlinkable address on the blockchain.17 This decouples the public identity from the on-chain assets.
The Protocol Flow (Alice to Bob):
1. Bob (Operator): Generates a Root Spending Key ($m$) and a Viewing Key ($v$). From these, he computes a Stealth Meta-Address ($M$) which essentially encodes the public keys corresponding to $m$ and $v$. This $M$ is public and can be shared freely.18
2. Alice (Donor): When Alice wants to donate, she uses Bob's $M$ to generate an Ephemeral Key ($r$) and computes a Shared Secret ($S$) using Elliptic Curve Diffie-Hellman (ECDH): $S = r \times V$ (where $V$ is Bob's viewing public key).
3. Alice: Computes a unique Stealth Address ($P$) using the shared secret $S$. The formula is typically $P = M + G \times \text{hash}(S)$.18 She sends the funds to this address $P$.
4. Alice: Publishes her Ephemeral Public Key ($R$) to an "Announcement" contract (ERC-5564 Announcer) on-chain. This announcement contains the data Bob needs to locate the funds.20
5. Bob (Extension): The browser extension scans the Announcement contract. For each announcement, it uses Bob's View Key ($v$) to attempt to "unlock" the Shared Secret ($S = v \times R$). If successful, it derives the private key for $P$ and claims the funds.21
Outcome: To an outside observer (e.g., a forensic accountant), the transaction looks like Alice sent money to a random, fresh address $P$. There is no on-chain link between $P$ and Bob's main accounts or his Meta-Address $M$.

The ESP32-S3 is selected for its native USB capabilities, allowing it to act as a USB composite device (CDC/HID) without external bridges.14 This enables direct communication with the Chrome browser via the WebUSB API.
Security Architecture:
1. Key Storage: The master private key (mnemonic) is stored in the ESP32's encrypted flash partition (NVS) or, ideally, a dedicated secure element like the ATECC608A if integrated into the PCB design.25 The key is encrypted at rest.
2. Air-Gapped Signing: The browser extension prepares an unsigned transaction (unsigned Tx). It does not have the key to sign it.
3. Transmission: The extension sends the RLP-encoded transaction (or its hash) to the ESP32 via WebUSB.16
4. Verification: The ESP32 firmware parses the transaction and displays the critical details (Amount, Recipient Address, Chain ID) on its local OLED screen (if equipped) or indicates them via LED codes.26 This creates a "What You See Is What You Sign" (WYSIWYS) environment, preventing the extension from tricking the user into signing a malicious transaction.
5. Physical Authorization: The Operator must physically press the "fidget" button (mechanical switch) on the device to authorize the signature.7 This is the "Human" component of the Centaur.
6. Signature Return: The ESP32 computes the ECDSA signature ($r, s, v$) using its internal crypto library (e.g., Web3E or trezor-crypto) and returns only the signature to the browser for broadcasting.27

	9.2 Firmware Command Protocol (WebUSB)
The communication between the Chrome Extension and the ESP32-S3 will utilize a simplified command-response protocol over the WebUSB interface.
* Command 0x01 (GET_INFO): Request device status, public keys, and firmware version.
* Command 0x02 (SIGN_TX): Send RLP-encoded transaction data. Device parses, displays, and waits for user confirmation. Returns signature ($r, s, v$) or error.
* Command 0x03 (SIGN_MSG): Send arbitrary message for EIP-191 signing. Used for proving ownership or logging into services.
* Command 0x04 (GET_ADDRESS): Request specific address derived from a BIP-44 path.
This protocol ensures that the "intelligence" of transaction construction remains in the flexible JavaScript layer (Extension), while the "wisdom" of authorization remains in the rigid Hardware layer (ESP32).
