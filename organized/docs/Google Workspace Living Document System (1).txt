3.2 Shadow IT Discovery and OAuth Token Forensics
Shadow IT in a modern Google Workspace environment rarely manifests as illicit software installed on a desktop; instead, it hides in the OAuth token layer. Users, often in a bid to increase productivity, grant third-party applications access to core data repositories like Drive, Gmail, or the Directory. These tokens represent persistent access keys that remain active even after the user has closed the browser, creating a massive, invisible attack surface known as "Ghost Apps".1
3.2.1 The Programmatic OAuth Audit Protocol
Standard reviews via the Google Admin Console (Security > API Controls) often fail to provide the granularity required for a deep forensic sweep. They present a summarized view but do not easily reveal the specific scopes granted by individual users or the timestamp of the last token refresh. To achieve a true forensic baseline, we must utilize a programmatic audit using the Admin SDK Reports API and the Directory API.1
The audit script functions by iterating through every user in the organizational unit (OU) structure via the AdminDirectory.Users.list method. For each user, it queries the tokens.list method, extracting critical metadata for every connected application: the client_id, displayText (application name), scopes (specific permissions granted), and the nativeApp status. This programmatic approach allows us to generate a comprehensive "Token Manifest" in BigQuery or Google Sheets.1
Risk Categorization Matrix:
* Tier 1: High Risk (Data Exfiltration Vectors): Applications requesting broad, sensitive scopes such as https://www.googleapis.com/auth/drive (Full Drive Access) or https://www.googleapis.com/auth/gmail.readonly. Any application with these scopes that is not on the corporate "Allowlist" is treated as a critical breach.
* Tier 2: Ghost Apps (Persistence Vectors): Applications authorized by users who are currently suspended, offboarded, or inactive. While the user cannot log in, the OAuth tokens may remain valid, providing a "backdoor."
* Tier 3: Phantom Tenants (Cross-Contamination): Extensions or apps that bridge the gap between "Corporate" and "Personal" Google accounts linked to the same Chrome profile.1
3.2.2 Remediation: The "Token Shred" Policy
Consistent with the "abdicate.sh" principle of key destruction, we implement a scripted remediation policy. This "Token Shred" protocol involves:
1. Allowlist Policy: Only explicitly vetted applications (e.g., Asana, Salesforce) are permitted.
2. Bulk Revocation: A script using AdminDirectory.Tokens.delete systematically revokes all tokens for non-allowlisted apps.
3. Future-Proofing: Configuring the "App Access Control" policy to "Block" all new third-party apps by default.1
